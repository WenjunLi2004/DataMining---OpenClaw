Label (target)is_top20 = 1 if current_stars ≥ 80th percentile of the sample (single-batch → global p80; multi-batch → per-batch p80). Stored in labels, never mixed into snapshot.
Anti-leakage design — every one of the 19 model features is derivable from the first-30-day window of a repository: commits / issues / PRs use GitHub API since + until filters; contributors_30d counts distinct commit-author identities within the window (login → email → name); README features are fetched at the most recent commit SHA ≤ created_at + 30 days. Current-state fields (author_followers, author_public_repos, has_license, current README, current contributors count, TF-IDF over topics / description) are deliberately excluded from the model. current_stars / current_forks live only in labels and never enter the feature vector.

Key Findings

LR achieves the highest AUC of 0.8830. All three models stay well above random, confirming that strictly first-30-day signals are predictive of long-term star growth.
The single strongest predictor is lang_Python (Gini 0.1611). Within the 19 strict-30d features, this is the dominant early signal of growth in this AI/ML repo cohort — interpret it as model attribution, not causal.
Adding historical README features (Group C) yields the largest ablation gain (+0.0186). Derived activity marginal contribution: -0.0072 (Group D vs C). Raw 30-day signals + README explain most of the gap.
Time-aware split is the deployment-style stress test: the chronological train→test AUC gap is bounded and stays close to random-CV AUC. Random 5-fold CV remains a reasonable proxy for this single-window dataset.

1. 实验结论

本次实验是一个历史回测，旨在利用仓库早期（前30天）的信号，预测该仓库最终是否能进入样本中按当前星标数排名的前20%（阈值：49星）。

  • 模型性能：逻辑回归（LR）在AUC（0.882965）和F1分数（0.63295）上表现最佳，同时在前10个预测中达到了80%的精确率（P@10=0.8）。随机森林（RF）在PR-AUC（0.661715）上表现最佳。
  • 超越基线：所有模型（AUC均>0.87）都显著优于最佳单特征基线（issues_30d，AUC=0.662836），证明了多特征机器学习方法的有效性。
  • 关键发现：在特征组消融实验中，加入“历史README（30天）”特征组（C组）带来了最大的AUC提升（+0.01864），表明README的早期内容对预测最终成功至关重要。
  • 时间稳定性：在时间分割验证中，随机森林模型的AUC（0.881569）与交叉验证结果（0.881216）高度一致，AUC差距仅为0.000352，表明模型在时间维度上具有一定的稳定性。

2. 模型行为解释

  • 逻辑回归 (LR)：表现出高召回率（0.841905）和中等精确率（0.513465），意味着它倾向于“捕获”更多潜在的热门仓库，但会包含一些误报。其P@10为0.8，表明在最高置信度的10个预测中非常可靠，适合用作候选名单筛选。
  • 随机森林 (RF)：表现出高精确率（0.631987）和低召回率（0.489048），意味着它预测为正例的样本有很高的概率是真正的热门仓库，但会遗漏很多潜在目标。其P@20为0.85，表明在Top-20推荐中非常精准，适合用于精选推荐。
  • XGBoost：性能介于LR和RF之间，在召回率和精确率上取得了平衡（F1=0.599388），P@10同样为0.8，也是一个可靠的候选模型。

总结：LR更适合“广撒网”式的候选名单生成，而RF更适合“精准打击”式的精选推荐。XGBoost则提供了一个平衡的选择。

3. 数据集问题诊断

  • 类别不平衡：正样本（is_top20）占比为20.4%（102/500），属于轻度不平衡。模型整体表现良好，但精确率-召回率权衡仍需关注。
  • 语言分布偏差：

- Python 的正样本率（62%）远高于全局平均（20.4%），表明Python仓库在样本中更容易成为热门。这可能导致模型过度依赖lang_Python特征，并可能降低其在其他语言上的泛化能力。 - TypeScript 和 Other 在样本中没有正样本（正样本率为0%），导致无法计算其AUC。这意味着模型对这两种语言的预测能力未知，结论不可靠。

  • 特征高度相关：多个特征之间存在强相关性，例如 commits_30d 与 activity_total_30d 的相关系数高达0.967666。这会导致特征重要性解释不稳定，模型可能将重要性分散到相关特征上，而非识别出真正的因果特征。
  • 标签定义：标签 is_top20 是基于仓库的当前星标数，这是一个历史回测标签，并非对未来成功的预测。实验结论不能直接等同于“发现未来的热门项目”。

4. 真正有意义的特征

综合Gini重要性、SHAP值和XGBoost重要性，以下特征被一致认为是重要的：

1. lang_Python：在所有重要性排名中均位列第一，是预测能力最强的特征。这与Python子集极高的正样本率（62%）相符。 2. activity_total_30d：早期总活动量是强有力的预测指标，在所有模型中重要性都很高。 3. issues_30d：早期Issue数量在SHAP和XGBoost重要性中排名靠前，且作为单特征基线时AUC最高（0.662836），表明它是一个稳健且独立的预测因子。 4. readme_len_30d：早期README的长度在Gini和SHAP重要性中均排名前五，并且消融实验证明加入README特征组带来了最大的性能提升，证实了其价值。 5. commits_30d：早期提交次数在Gini重要性中排名第三，是衡量早期开发活跃度的核心指标。

注意：lang_JavaScript 和 lang_TypeScript 在Gini重要性中排名靠前，但鉴于其正样本率极低或为零，其重要性可能反映了模型学习到了“非Python”的模式，而非其本身是成功的驱动因素。

5. 不能过度相信的结论

  • “模型能预测未来的热门项目”：不能。这是一个历史回测，标签是“当前星标数”，而非未来的星标数。模型学到的是早期特征与当前成功之间的相关性，而非因果关系。
  • “TypeScript/Other语言的预测是可靠的”：不能。由于样本中这两个语言的正样本数为0，模型无法学习其模式，任何相关预测都是不可信的。
  • “commits_30d 比 activity_total_30d 更重要”：不能。这两个特征高度相关（r=0.967），重要性排名可能因模型和随机性而波动。更合理的解释是“早期开发活跃度”这个整体概念很重要。
  • “Python仓库更容易成功是因为其早期行为”：不能。Python子集的高正样本率可能源于样本选择偏差（例如，样本中Python项目本身质量更高），而非其早期行为模式。模型可能只是学到了这种偏差。
  • “AUC 0.88 意味着模型非常完美”：不能。虽然AUC很高，但精确率（特别是LR的0.513）表明仍有大量误报。在Top-10/20的精确率（0.7-0.85）才是更贴近实际应用场景的指标。

6. 下一步实验建议

1. P0: 验证特征数据质量：确认所有500个样本的特征都是通过“严格30天收集器（v2+）”获取的。当前数据表明所有特征都是严格30天的，但建议在数据管道中增加自动化校验。 2. P1: 进行“Today Radar”模式实验：设计一个实验，使用最新的仓库数据（而非历史数据）作为输入，预测其未来（例如，未来6个月）的星标增长。这才能验证模型在真实世界中的前瞻性。 3. P1: 处理语言偏差： - 对Python子集进行单独建模或分析，以理解其高正样本率的根本原因。 - 收集更多TypeScript和其他语言的样本，特别是正样本，以提升模型的泛化能力。 4. P2: 特征工程与降维： - 考虑合并高度相关的特征（如 commits_30d 和 activity_total_30d），或使用PCA等降维技术，以提高模型的可解释性和稳定性。 - 探索更多有意义的早期特征，例如：README中是否包含特定关键词（如“AI”、“tool”）、早期Star的增长速度、是否有外部链接等。 5. P2: 模型集成：考虑将LR（高召回）和RF（高精确率）进行集成，构建一个既能保证覆盖率又能保证精确率的混合模型。

7. 可追问问题

1. 在“Today Radar”模式下，我们如何定义和获取“未来成功”的标签？例如，是使用未来6个月的星标增长率，还是达到某个绝对星标数？ 2. 对于Python子集极高的正样本率（62%），我们能否获取更多关于这些Python仓库的元数据（如主题、所属组织类型）来分析其共性？ 3. 我们是否有计划为TypeScript和其他低样本语言补充数据？如果需要，数据采集的优先级和预算如何？ 4. 在特征工程中，我们是否可以尝试构建一些比率特征（如 issues_per_contributor_30d）来替代高度相关的原始计数特征，以提高模型鲁棒性？ 5. 对于“Today Radar”的候选名单，业务上期望的精确率和召回率目标是多少？这将决定我们最终选择哪个模型或阈值。

⚠️ Finding #1: Sampling Policy Contamination

The ts_broad strategy — designed to fix TypeScript underrepresentation — introduced a systematic bias that propagates through the entire model.

What happened

DimensionN500N1500
TypeScript sample count99231
TypeScript % of dataset6.2%15.4%
TypeScript positive rate (is_top20)~20%65.4%
TypeScript AUCN/A (insufficient samples)0.6493
#1 RF Gini featurereadme_len_30d (historically 0.2833)lang_TypeScript (0.1730)

Root cause: three-way confound

  1. ts_broad removes the AI-keyword filter for TypeScript, pulling in any TypeScript repo with 5–500 stars — a much broader population than the AI-focused Python/Go/Rust samples.
  2. Early cohorts (early1 03~04, early2 01~02) have 17–19 months of star accumulation by the observation date, inflating absolute star counts vs the 11-month mid cohort.
  3. Batch-wise p80 labelling sets a higher threshold per batch (336 and 323 stars vs 48), but the TS repos — pulled from the long-tail of any TypeScript project — happen to cluster above these thresholds at a 65% rate.
Consequence: lang_TypeScript became the #1 Gini and SHAP feature in N1500 (importance 0.1730), displacing readme_len_30d (historically 0.2833 in N500). This is a statistical artifact of the sampling policy, not a causal signal. In a real deployment, predicting that "TypeScript repos grow faster" would be wrong — it is the population of TS repos sampled that happens to have more stars, not TypeScript itself.
⚠️ Finding #2: Cross-Batch Label Distribution Shift

Batch-wise p80 labelling was designed to make each cohort's positive rate ~20%. It achieves this locally, but the absolute star counts defining "success" differ by 7× across batches — the same label token means completely different things.

p80 threshold by batch

Why this matters: a model trained on all three batches jointly learns to predict "is this repo in the top-20% of its own age cohort?" — which is a different task depending on the batch. A repo with 100 stars is "Top-20%" in the mid cohort but a clear failure in early1/early2. The label is nominally the same but semantically different. Batch-wise p80 can equalise positive rates but cannot equalise the underlying growth dynamics between cohorts created months apart.
⚠️ Finding #3: k-fold CV Overestimates Performance on Multi-Batch Data

Standard k-fold CV randomly shuffles all samples across folds, mixing repos from different time periods and star-accumulation stages. When data spans multiple non-exchangeable cohorts, CV optimistically overestimates real-world performance.

What the gap means: for N1500, the random k-fold CV reported AUC 0.9066, but when the model was actually asked to predict on later-created repos (chronological split), AUC dropped to 0.8126 — a 9.4% overestimation. This happens because CV folds contain repos from different batches mixed together; the model sees early1/early2 repos in both train and test sets, and the label distribution shifts between batches give it an artificial advantage.

N500's gap is only +0.0025 because all 500 repos come from the same two-month window and are therefore approximately exchangeable across time. This confirms that single-window datasets are not just simpler — they are more methodologically sound for CV-based evaluation.

Discussion: Implications for Open Source Success Prediction

What this study shows

1. Naive dataset scaling is not equivalent to better science. Adding more data from adjacent time periods improved raw AUC numbers (+0.03–0.05) but introduced three distinct methodological problems that make the N1500 results harder to interpret and less actionable than N500.
2. Sampling policy decisions cascade into model behaviour. The ts_broad fix for TypeScript underrepresentation — sensible in isolation — combined with longer star-accumulation time to produce a spurious feature signal. Data collection and model evaluation are not independent stages.
3. Label definition must be absolute, not relative-to-cohort. Batch-wise p80 labelling trades one problem (imbalanced positive rates) for another (semantically inconsistent labels). For multi-cohort studies, a fixed absolute threshold (e.g., 100 stars regardless of cohort) or a growth-rate label (stars/month) would be preferable.
4. Standard k-fold CV is unreliable for multi-batch data. The N1500 time-split gap of −0.094 means real-world performance would be dramatically lower than what CV suggests. Blocked CV (never mixing cohorts in the same fold) or leave-one-cohort-out CV would be more honest.

Future work

Recommended directions: (a) Use a fixed absolute star threshold or log-normalised regression target; (b) Apply blocked cross-validation with cohort as the blocking variable; (c) Replace ts_broad with domain-scoped TS queries (AI-specific TS frameworks); (d) Collect longitudinal data (same repos at 3m, 6m, 12m) to study growth trajectories rather than cross-sectional snapshots; (e) Explore embedding features from DeepSeek API as a future extension — current model already excludes bag-of-words TF-IDF to enforce strict 30-day boundedness.
