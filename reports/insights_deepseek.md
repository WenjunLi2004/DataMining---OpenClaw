## 1. 实验结论
本次实验基于 500 个仓库的 30 天早期窗口特征，以当前星标数排名前 20%（阈值 49 星）作为分类目标。
- 最佳 **AUC** 为 **RF 模型 0.8879**（标准差 0.0223），最佳 **PR-AUC** 同为 RF 模型 0.6517（标准差 0.0867）。
- 单指标最佳表现分散：**LR 模型 F1 最高 0.6330**（召回 0.8419，精确 0.5135），**P@10 最佳 0.80** 出现在 LR、RF、XGBoost 三模型中，**P@20 最佳 0.80** 由 XGBoost 达成。
- 与单特征最强基线（issues\_30d, AUC 0.6628）相比，RF 模型 AUC 提升 0.2251；与随机排名基线（P@10=0.204）相比，模型 P@10 提升幅度可观。
- 消融实验揭示：仅含语言和仓库主特征的 A 组 AUC 为 0.8557，加入 30 天活动特征后提升最大（+0.0160 AUC），加入历史 README 特征再次提升 0.0149 AUC，而派生特征仅贡献 0.0013 AUC。
- 时间分割验证（训练 400，测试 100，分割日期 2025-04-18，测试正例率 0.15）下 RF 模型 AUC=0.8831，与随机 CV 的 AUC 差距仅 -0.0048，模型稳定性表现良好。

## 2. 模型行为解释
- **高召回、中精度模式**：最优 F1 的 LR 模型精确 0.5135，召回 0.8419，呈现“宁可多报、不愿漏报”的模式，适合在候选初筛场景使用。RF 模型在召回 0.7029 时精确 0.5846，更均衡。
- **活动信号是核心驱动力**：消融实验中加入早期活动特征使 AUC 从 0.8557 升至 0.8717，是全过程中最大的一次信息增量。说明早期代码提交、issue 等活动的密度对区分头部项目有实际帮助。
- **README 特征提供中等增量**：30 天内的历史 README 内容、长度等信息使 AUC 增加 0.0149，表明文档建设在早期信号中有一定解释力，但不如活动特征突出。
- **派生特征几乎不贡献**：从 C 组到 D 组仅提升 0.0013 AUC，说明基于活动特征衍生出的复合指标（如人均提交量、PR 占比等）在当前模型中提供了冗余信息，未明显增加区分力。

## 3. 数据集问题诊断
- **Python 子集正例率显著偏离全局**：Python 100 个样本中正例率高达 0.62，超出全局正例率（0.204）0.416。这可能导致模型学习到语言代理而非通用信号——在 Python 仓库中“盲目”倾向判为正例即可获得较高准确率。Python 的 per-language AUC 仅为 0.6744，低于有可比 AUC 的语言平均值 0.130（该均值排除无 AUC 的语言后为 0.7740），验证了这一结论。
- **TypeScript 无正例**：TypeScript 的 99 个仓库中正例数为 0，模型无法训练二分类器，因此 per-language AUC 无法计算。JavaScript 仅有 2 个正例（正例率 0.02），虽然 AUC 为 0.8878，但极端不平衡的样本使其统计结论需谨慎对待。
- **特征间高度共线**：commits_30d 与 activity_total_30d 相关性达 0.9677，commits_per_contributor_30d 与两者相关性均高于 0.76。强相关意味着特征重要性被分散或不稳定；无法将模型收益归因于单一特征。
- **样本量限制 per-language 分析的可靠性**：Other 语言只有 3 个样本（正例为 0），完全缺乏代表性。Go 语言 AUC 仅 0.6488 且标准误达 0.1528，表明模型在 Go 子集中表现不稳定。

## 4. 真正有意义的特征
1. **lang_Python（Gini 重要性 0.186，XGBoost 重要性 0.327）**：语言类别特征在树模型中的主导地位说明当前建模中标签分布影响了特征权重。这不是“Python 语言更好”的证据，而是样本选择的反映。
2. **commits\_30d（Gini 0.164）** 与 **activity\_total\_30d（Gini 0.102）**：代表早期开发密度的两个高影响力特征，但强相关意味着二者携带类似信息；消融实验也证明活动组块贡献明确。
3. **lang\_JavaScript / lang\_TypeScript（Gini 约 0.100/0.086）**：与 Python 类似，反映特定语言子集的标签分布特征，而非真实区分力。
4. **issues\_30d（Gini 0.071，XGBoost 0.050）**：作为独立单特征排序器已有 AUC 0.6628，并且在模型特征重要性中保持稳定存在。与其正相关的 prs\_30d 以及复合特征 prs\_per\_issue\_30d 也进入重要性列表，进一步强化“互动密度”信号。
5. **readme\_len\_30d（Gini 0.072）**：单特征 AUC 0.5602，P@10 为 0.50，P@20 为 0.30，能力中等，但在模型中持续提供增量信息（消融实验 C 组加入 README 相关特征提升 AUC 0.0149）。

## 5. 不能过度相信的结论
- **RF AUC 0.8879 ≠ 实时预测今日未来热门的能力**：当前任务是带有 look-ahead bias 的历史回测（使用 current\_stars 做标签）。实时部署无未来标签，“预测”本质为候选筛选，不保证能准确复现 0.88 AUC。
- **“语言驱动模型”不代表语言本身有因果效力**：Python 的高正例率使模型容易学习到“Python 约等于前 20%”的捷径，导致模型在非 Python 子集的表现被高估或低看。不能理解为 Python 项目确实更优。
- **JavaScript、TypeScript 的准确度和重要性结论不可信**：JS 正例仅 2 个，仍产生 AUC 0.888，TypeScript 无法测算 AUC。这类小样本结论不能推广。
- **不要对相关特征作因果解读**：commits\_30d 和 activity\_total\_30d 相关性 0.968，commits\_per\_contributor\_30d 与两者的相关性均超过 0.76，意味着排名特征之间信息重度冗余。特征重要性来自模型内部的使用偏好，不能推断哪个有独立因果贡献。
- **验证差异虽然小，不代表完全稳健**：时间分割测试集仅 100 个样本，正例率 0.15，P@20 下滑到 0.50，显示模型在更近期、更低正例率的分布上排序效果有所减弱。需更多时间切面验证。

## 6. 下一步实验建议
1. **移除或显式控制 Python 样本的过代表**：重采样或加权训练，检验 AUC 及 PR-AUC 是否下降、哪些特征稳定性变化，以量化语言捷径的影响。
2. **增加单特征简单排序器的对比**：当前报告里 issues\_30d 表现最好（AUC 0.663，PR-AUC 0.360，P@10=0.60），应定期产出“简单启发式 vs 复杂模型”的基线表，以监控模型增值是否在缩小。
3. **精简特征集合**：剔除 commits\_30d 或 activity\_total\_30d 其中之一、以及与 issues 高度相关的冗余变量，测试对模型泛化能力的影响。当前消融实验显示从 15 个特征到 19 个特征收益仅 0.0013 AUC，暗示特征可减法。
4. **严格时间泛化验证**：在多个不同的 snapshot 时间点（不只一个）进行 train/test 切割，检查 AUC 和 P@K 的波动范围是否受控，并报告 95% 置信区间；目前的单一切割不足以支撑“模型随时间泛化良好”的强结论。
5. **针对 TypeScript、JavaScript 扩充样本**：如要维持 per-language 分析，TypeScript 需要能构建正例的最小样本量（建议 >=50 正例），JS 目前 2 正例不具备分析条件。

## 7. 可追问问题
1. “Python 子集内的高正例率是否反映了真实数据规律，还是样本抓取偏差？如果在 Python 样本中随机下采样使正例率回到 0.20，模型的主要特征排序和 AUC 会发生多大改变？”
2. “单一排序特征（如 issues\_30d 的 AUC 0.663、P@10 0.60）在实际候选筛选任务中是否已经足够好，值得在展示 Top 10 列表时直接使用？模型额外增益（AUC +0.225）是否能转化为 P@K 的清晰提升？”
3. “若将未来标签从 current\_stars 换成 360 天后观测到的星标增量，模型在相同特征下的 AUC 和 P@10 还能维持当前水平吗？这是否才是用户真正关心的雷达模式？”
4. “README 的哪些细分信号（长度？图片？外链数？）贡献了从 B 组到 C 组 AUC 的 +0.0149 提升？是否能进一步拆解为更稳定的可解释指标？”
