## 1. 实验结论
- 在 500 个历史仓库的 30 天早期窗口上，**RF 模型**以 **AUC = 0.887922** 成为最佳单一模型。  
- 所有模型在 **precision@10 均达到 0.8**，显著优于随机基线（0.204）。  
- 全局正例率 20.4%，Python 子集高达 62%，TypeScript 子集为 0%，语言分布极不平衡。  
- 时间外验证（2025-04-18 之后 100 个仓库）RF 的 AUC 为 0.883137，与交叉验证差距仅 -0.004784，未见明显过拟合。  

## 2. 模型行为解释
- **召回优先**：LR 召回 84.2%，但精度仅 51.3%，适合“候选池”场景。  
- **平衡取舍**：RF 召回 70.3%，精度 58.5%，整体 F1 与 AUC 更均衡。  
- **特征贡献**：  
  - Gini 与 XGBoost 重要性均把 **lang_Python** 排在首位（0.186 / 0.327）。  
  - 30 天活动总量类特征（commits_30d、activity_total_30d）紧随其后。  
- **相关性陷阱**：commits_30d 与 activity_total_30d 相关系数 0.968，解释时需避免重复归因。  

## 3. 数据集问题诊断
| 问题 | 证据 | 影响 |
|---|---|---|
| 语言分布不均 | Python 正例 62/100，TypeScript 0/99 | TypeScript 结论不可信 |
| Python 子集标签漂移 | 正例率 +41.6% vs 全局 | 拉低该语言 AUC（0.674） |
| 特征多重共线性 | commits_30d ↔ activity_total_30d (0.968) | 特征重要性需用 SHAP 校正 |
| 样本量小 | Other 语言仅 3 例 | 统计功效不足 |

## 4. 真正有意义的特征
1. **lang_Python**（跨模型最高重要性）  
2. **commits_30d**（早期开发节奏）  
3. **issues_30d**（单特征 AUC 0.663，为基线最佳）  
4. **activity_total_30d**（综合活跃度代理）  
5. **readme_len_30d**（文档投入度，与 commits 仅 0.215 相关，独立信号）

## 5. 不能过度相信的结论
- **TypeScript 零正例** ⇒ 不能得出“TypeScript 不会出热门仓库”的结论（样本仅 99）。  
- **Go AUC 最低（0.649）** ⇒ 但正例率与全局接近，可能是特征在该语言下区分度不足，而非语言本身劣势。  
- **Gini 重要性** ⇒ 连续变量被高估，需结合 SHAP 解释因果。  
- **Today Radar 实时预测** ⇒ 当前标签为历史 backtest，不能直接外推未来。  

## 6. 下一步实验建议
1. **P0** 维持历史 backtest 作为课程评估基准，避免标签泄漏。  
2. **P1** 收集更多 TypeScript 与 Other 语言样本，减少语言偏差。  
3. **P1** 用 SHAP 重新计算全局与 per-language 特征贡献，校正共线性影响。  
4. **P2** 设计“今日雷达”轻量模式：  
   - 仅输出候选短名单，标注置信度；  
   - 用滚动 30 天窗口持续收集新仓库，待 6 个月后回填真实标签再评估。  
5. **P2** 尝试对 Python 子集单独训练，校准模型，处理标签漂移。
