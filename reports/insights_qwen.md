## 1. 实验结论

- **任务定义**：基于仓库前 30 天的早期信号，预测其是否能进入当前 star 数的前 20%（阈值设定为 49 stars）。
- **数据规模**：总样本量为 500，其中正样本 102 个，正样本率（positive_rate）为 0.204。
- **模型表现**：随机森林（RF）模型综合表现最佳，AUC 达到 0.887922，PR-AUC 为 0.651676。逻辑回归（LR）模型取得了最高的 F1 分数（0.63295）和召回率（0.841905）。所有三个模型（LR、RF、XGBoost）的 P@10 均达到了 0.8。
- **基线对比**：最佳单特征基线 `issues_30d` 的 AUC 为 0.662836，P@10 为 0.6。多特征机器学习模型显著优于单特征启发式排序（随机基线 P@10 仅为 0.204）。
- **时间泛化能力**：在时间切分验证中（训练集 400，测试集 100，切分日期 2025-04-18），RF 模型的测试集 AUC 为 0.883137，与随机交叉验证 AUC（0.887922）的差距（auc_gap）仅为 -0.004784，表明模型在时间维度上未出现明显过拟合。

## 2. 模型行为解释

- **特征组贡献（消融实验）**：早期活动特征（Early Activity）和历史 README 特征对模型性能提升起主导作用。从基础特征（A组，AUC 0.855715）加入早期活动特征（B组）后，AUC 提升了 0.015983；加入 README 特征（C组）后，AUC 提升了 0.014904。而加入所有 19 个特征（D组）仅带来 0.00132 的微小提升，说明核心预测能力已在前 15 个特征中饱和。
- **模型权衡（Tradeoffs）**：LR 模型呈现“高召回、低精确”的特点（召回率 0.841905，精确度 0.513465），非常适合用于初步的候选短名单（shortlist）筛选；RF 和 XGBoost 则在精确度和召回率之间更为平衡，RF 的精确度达到了 0.584595。
- **模型归因**：在 RF（Gini 重要性）和 XGBoost 中，`lang_Python` 均被列为最重要的特征（重要性分别为 0.1862 和 0.3270），其次为 `commits_30d`、`activity_total_30d` 等早期活动指标。

## 3. 数据集问题诊断

- **类别分布偏移（Python 异常）**：Python 子集的正样本率高达 0.62（62/100），远高于全局的 0.204。这种极端的类分布偏移和语言内部同质性导致 Python 子集的 AUC 偏低（0.674416）。
- **单类问题（TypeScript 异常）**：TypeScript 子集（99 个样本）的正样本数为 0（正样本率 0.0），导致无法计算该语言的 AUC（标记为 single_class）。
- **特征多重共线性**：数据中存在强相关特征对。`commits_30d` 与 `activity_total_30d` 的相关系数高达 0.967666，`commits_30d` 与 `commits_per_contributor_30d` 的相关系数为 0.820285。这会干扰基于树模型的特征重要性解释。
- **数据收集验证**：当前数据声明所有 500 行均使用了严格的 30 天收集器（`all_strict_30d: true`），README 和 contributors 的严格 30 天计数均为 500，近似计数为 0，数据血缘（provenance）符合预期。

## 4. 真正有意义的特征

- **核心预测特征**：结合消融实验与特征重要性，`lang_Python`（语言特征）、`commits_30d`（30天提交数）、`activity_total_30d`（30天总活动量）以及 README 相关特征（如 `readme_len_30d`）是对预测 Top 20% 真正有意义的特征。
- **最佳单特征**：`issues_30d`（30天 issues 数）是最佳单特征基线，AUC 达到 0.662836，P@10 达到 0.6，显著优于其他单特征（如 `readme_len_30d` AUC 0.560203，`commits_30d` AUC 0.555929）。
- **未测量的特征解释**：当前数据没有提供 SHAP 值（`top10_shap` 为空），也没有提供 LR 的绝对系数（`top10_lr_abs_coef` 为空），因此无法从 SHAP 边际贡献或线性权重的角度进一步验证特征的真实重要性。

## 5. 不能过度相信的结论

- **语言级别的泛化结论**：由于 TypeScript 正样本为 0，JavaScript 正样本仅 2 个（正样本率 0.02），且整体样本量较小（N=500），不能过度相信针对特定语言（尤其是 TS 和 JS）的模型表现和 AUC 结论。
- **特征重要性的因果性**：由于 `commits_30d`、`activity_total_30d` 和 `commits_per_contributor_30d` 之间存在高度相关性，且 Gini 重要性容易过度加权连续数值特征，当前的特征重要性仅代表模型归因，不能解释为因果归因。
- **实时发现未来爆款的能力**：当前任务是历史回测（historical_backtest），标签 `current_stars` 是历史状态。这不能证明模型具备实时发现“今天”未来爆款的预测能力。

## 6. 下一步实验建议

- **P0**：保持默认实验作为固定的历史回测，以避免不稳定的实时标签，确保课程评估的可重复性。
- **P0**：如果将模型应用于“Today Radar”模式，必须将其定位为“候选短名单（candidate shortlist）”而非经过验证的预测，因为近期仓库尚未产生未来标签。
- **P1**：将模型排序结果与简单的单特征基线（如 `issues_30d`）进行直接对比分析，以量化机器学习管道相对于简单启发式规则的具体增益。
- **P1**：验证原始数据是否确实由 strict-30d 收集器（v2+）收集，确保 19 个特征严格限制在前 30 天内，避免近似值降低模型有效性。
- **P2**：在收集更多样本之前，对 TypeScript 和按语言细分的结果保持谨慎，并在报告中添加相应的免责声明。

## 7. 可追问问题

- 当前数据没有提供 SHAP 值，是否可以补充计算 SHAP 值以更安全、准确地解释特征重要性，从而消除 Gini 对连续数值特征（如 `commits_30d`）的偏好？
- Python 子集的正样本率（0.62）为何如此之高？在数据采样阶段是否引入了针对 Python 的偏差，或者这反映了特定时间段内 Python 仓库的极端头部效应？
- 对于 TypeScript（0 正样本）和 JavaScript（2 正样本）子集，是否需要调整采样策略或引入分层抽样来改善类别不平衡问题？
- 在时间切分验证中，测试集的正样本率下降到了 0.15（全局为 0.204），这是否意味着近期仓库的整体 star 增长趋势放缓，或者 49 stars 的阈值在近期变得难以达到？
